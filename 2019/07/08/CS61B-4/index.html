Generics and Autoboxing

  • Automatic Conversions

    • Autoboxing and Unboxing
      Cannot provide primitive type as an actual type argument
      Use corresponding reference type (wrapper classes)
      Java implicitly convert between primitive and wrapper types
      Arrays are never auto-boxed or auto-unboxed
        e.g. Integer[] x = (int[]) x; (compiler not allowed)
      autoboxing and unboxing has a measurable performance impact wrapper types use much more memory than primitive types
        every object has a 64 bit refetrence variable and requires 64 bits of
        overhead to store things like the dynamic type of the object
    • Widening
      Java automatically widen a primitive if needed
      Variable of type T1 will be implicitly cast to type T2 which takes a wider range of values than T1 (e.g. int => double)
      Have to manually cast when going from a wider type to a narrower type
        e.g. double x = 20; int y = (int) x;

  • Immutability
    immutable data type: cannot change in any observable way after instantiation
      e.g. String => remain unchanged after calling any method on String
    any data type with non-private variables is mutable unless declared final
    (final: prevent the variable from being changed after first assignment)

    • Pros
      prevent bugs and makes debugging easier (properties cannot change)
      can count on objects to have a certain behavior/trait
    • Cons
      need to create a new object in order to change a property
      Declare a reference as final does not make the object that reference is pointing to immutable
      Reflection API possibly change even the private variables

Exceptions

Exception breaks normal flow of control, handling error.

  • Throw Exceptions
    Exceptions are objects in Java
    throw new ExceptionObject(para1,…)
  • Catching Exceptions
    try and catch breaks the normal flow of the program
      protect program from exceptions
      corrective action
      keep error handling separate from the rest of the program
  • Uncaught Exceptions
    Exception thrown => descends the call stack => bottom of the stack => crash and print out the stack trace

  • Checked vs. Unchecked Exceptions
    • Compilation Error
      Exception must be caught or declared to be thrown
       => Compiler thinks some exceptions must be handled
    • Checked Exceptions
      Must be handled
      2 ways to handle checked error
       Catch: handle it right now
       Specify: defer handling & mark methods as dangerous
    • Unchecked Exceptions
      Not need to be handle and compile just fine but may crash at runtime
      Include Error & RuntimeException (unknown till runtime & unrecoverable)

Object Methods

All classes extend from the Object class.

  • toString()
    gives String representation of the object
    System.out.print() implicitly calls the method
    By default, Object class prints out the address of the object (hexadecimal String)
    String ‘+’ operator return a new String each time which take time linearly in the length of the whole String
      => use StringBuilder which is mutable
  • equals(Object o)
    • == operator
      == checks if two variables hold the same bits
      if equal for primitive and if address equal for reference
    • equals(Object o)
      By default, it is the same as == if not overriden
      Checks if null or other class (.getClass()) or the same address (i.e. the same object)

Packages and Access Control

  • Packages
    • Package - namespace organizing classes and interface
      Issue: multiple classes with the same name
      Solution: Appending class with a website address(backwards)
         e.g. ug.joshh.animal; // website is josh.ug
    • Using Packages
      1. Access within the same package
        Use the class name
      2. Outside the package
        Use entire canonical name or import package and use simple name
        Wildcard import
          e.g. import ug.joshh.animal.*
          //danger: conflicts for multiple packages with same class
    • Creating a Package
      1. Put the package name at the top of each file
      2. Store the file in the folder that has the appropriate folder name.
        e.g. ug.joshh.animal => ug/joshh/animal folder
      • Create a Package in IntelliJ
        File -> New Package -> choose package name
      • Add New Java Files to a Package in IntelliJ
        Right-click package name -> New -> Java Class
      • Add Old Java Files to a Package in IntelliJ
        Add “package [name]” to the top of file -> move .java file into folder
    • Default Packages – class with no package name at the top
  • JAR Files
    Zip all files by creating a JAR file (contain all .class file with additional info)
    JAR file is not safe => possible to unzip back into .java files
    • Creating a JAR File in IntelliJ
      1. File -> Project Structure -> Artifacts -> JAR -> “From modules with dependencies”
      2. Ok multiple times
      3. Build -> Build Artifacts (create a JAR file in a folder called “Artifacts”)
    • Build Systems – Automate the setting up of project structure
      Maven, Ant, Gradle
    • Access Control
      Package Private (black box)
        default access given to Java memebers without explicit modifier
        accessible between the same package (including default package)
      Public
        Signature should not change once deployed, if developers want to   cancel public things, they call it deprecated
      Subtleties
        Access is based only on static type
        Default access mode for Interface is public not package private
