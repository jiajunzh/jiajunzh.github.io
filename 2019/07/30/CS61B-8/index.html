CS61B-8:Hashing, Heap, Priority Queue

Posted on 2019-07-29 | Post modified: 2019-08-04 | In CS61B , Lecture Notes

Hashing

  • Issue with the Former Data Structure
    1. Items need to be comparable
    2. Can do better than θ(logN)
  • Data Indexed Integer Set
    Improve complexity from θ(logN) to θ(1)
    • Implementation
      Constructor: large ArrayList of type boolean
      add(int x): set x position to true
      contains(int x) return whether x position is true
    • potential issues
      ☑️wasteful space
      ☑️insertions of other type
  • Data Indexed Word Set
    • Big Idea
      String =(map)=> some Integer => Data Indexed Integer Set
    • App #1 - Use the first letter
      Collision: two different inputs mapped to the same one
    • App #2 – Avoiding Collisions
      String to unique Base26 Integers
        e.g. “cat” = 3*26*26 + 1*26 + 20*1 = 2074
      ☑️Still wasteful of memory
      ☑️Overflow if support Strings beyond single English words
  • Hash Codes - Take an object and convert it into some integer
    • App #1 - Default .hashCode() method using memory address of object
    • App #2 - Write our own hashcode method
    • Properties of HashCodes
      1. Must be an integer
      2. .hashCode() return the same number each time
      3. Two objects that are .equal() must have the same hash code
      Good hashCode distribute items evenly
  • Collision Handling
    • Solution: array contain a LinkedList of items instead of just items
    • Implementation
      • Nothing at index
        create a new LinkedList, place it and add new item
      • Index Already taken
        simply add item to list
        No duplicate keys => must check if key exist already
        Insert to the end of list => have to check all the elements anyways
    • Workflow
      • add item
        get hashcode of item
        if index has no item => create new list and place item there
        if already a list => check if the key exists and add item to list
      • contains item
        get hashcode of item
        if index is empty => flase
        otherwise, check all items in the list at that index
    • Runtime complexity of worst case
      contains - θ(Q): have to look at all items in list
      add - θ(Q): have to make sure the item not in the list yet
    • Space solving
      Modulo – hashcode % size of ArrayList
      LinkedList expected to be longer
  • HashTable
    • Workflow
      Input => hashcode => modulus to get index
    • Dynamically growing our hashtable
      M buckets and N items => Load Factor L = N/M
      • Solution – Set load factor threshold
        Create a new HashTable with 2M buckets
        Iterate thru all items in the old HashTable, add them into new HashTable
        (modulus changes => may into different buckets)
      • Runtime Complexity
        add and contains
          θ(L)= θ(1) => assuming items evenly distributed
        Resizing
          Already know no duplicates => add to the front of LinkedList
          Overall θ(N) => each items need θ(1)
    • Improve our hashcodes
      Giving random values for different items => evenly distributed
      • Strategies
        Use a base strategy in which the base is a prime
        ☑️Too small => String with small length spread within small ranges
        ☑️Too big => Caused by collision due to overflow
          e.g. base126 (string with same last 32 chars => same hashcode)

Heaps and Priority Queue

  • Priority Queue Interface
    Quickly find the smallest or largest item
    Space/memory benefits to using PQ => if only need the top N

  • Heaps

    • Min-heap Definition
      Min-heap property: each node <= its children
      Complete: missing item only at bottom level, all nodes as left as possible
    • Heap Operations
      • add: add to then end of the heap temporarily. Swim up to the proper place.
      • getSmallest: return the root of the heap
      • removeSmallest: Swap the last item in the heap into the root. Sink down to the proper place
    • Tree Representation of Heap
      • App #1a: Fixed-Width node => pointers to Children
         
      • App #1b: Variable-Width => use of array
         
      • App #1c: Variable-Width => reference to siblings
         
      • App #2: Store a key and a parent array
        Tree is complete
        parent array is redundant
        The order of reading the key array is the same as the level-order
         
      • App #3: Represent by level order in an array
        Implementation of Heap
        leftChild(k) = k * 2, rightChild(k) = k * 2 + 1, parent(k) = k / 2
         

  • Performance Analysis
    add - θ(log⁡N )
    getSmallest - θ(1)
    removeSmallest - θ(logN)
    Also solve the problem of duplicate elements
    Heap operations are amortized => resizing
    Array-based heaps take arount 1/3rd the memory as App #1a

  • Remaining issues
    1. How to order the items (comparator)
    2. How to make MinPQ into MaxPQ
